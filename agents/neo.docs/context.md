- **Sprint 4 Phase 1 implemented (2026-07-14, `*swe impl phase-1`):** new `src/scalene/scanner.py` — `Scanner` Protocol, frozen `Resource`/`ScanResult` dataclasses, `FileScanner`, `URLScanner`, `SCANNERS` registry (`docs/ARCHITECTURE.md` §13.2). Key implementation choices not fully spelled out in the arch doc, decided here:
  - **Generic fallback scope**: applies to every string value in `args` (not gated to any specific tool), which is what makes `Bash`'s `command` field "wired in" — no Bash-specific code path exists, it's just one more string that both scanners' fallback regexes see. Same mechanism also means a `WebFetch` `prompt` or any future tool's stray string field gets scanned too, for free.
  - **Path fallback regex is deliberately conservative**: only matches tokens with an explicit `/`, `./`, or `../` prefix (`_PATH_FALLBACK_RE` in `scanner.py`) — a bare relative token like `src/foo.py` (no prefix) does **not** match. Chose precision over recall here since an unprefixed word-with-slashes is much more likely to be a false positive (git refs, ratios in prose, etc.) in a generic string-scanning context than in a known path field. Flagging in case Trin's UAT wants broader relative-path coverage — easy one-line regex change if so.
  - **URL fallback captures host directly** via a named `(?P<host>...)` group stopping at `/`, whitespace, `:`, or quote chars — used identically for both the known `WebFetch.url` field and the generic fallback (one regex, one code path, not two different host-extraction strategies), so a URL found via either route always normalizes to the same host string. Resource identity is the host, never the full URL/path, per §13.1.
  - **`scan()` reuses `subprocess_isolation.run_scanner` unchanged** — `FileScanner.scan()` calls `run_scanner("secrets", ...)`, `URLScanner.scan()` calls `run_scanner("reputation", ...)`, preserving the existing STORY-601 isolation boundary exactly. `ok=True` → `"public"`/`"trusted"`, `ok=False` → `"sensitive"`/`"untrusted"` with `run_scanner`'s `reason` passed through. Deliberately did **not** try to distinguish "scan found something" from "scan machinery failed" at this layer — `run_scanner` already collapses both into `ok=False`, and untangling that is explicitly Phase 4's job (STORY-1004 fatal-exit boundary, `cli.py`), not Phase 1's. Flagged so Phase 4 doesn't assume `FileScanner`/`URLScanner` already do this distinction.
  - Known-field identity normalization: file paths go through `os.path.abspath()` (works on non-existent paths, no filesystem I/O, unlike `.resolve()`) so a relative `file_path` still produces a stable absolute cache-key-shaped identity without requiring the file to exist yet — matters because `pre_tool_use` will eventually call `identify()` before a `Write` has actually created the file.

## Key Findings
- `jsonpath_ng.parse` (not `.ext`) is sufficient for all STORY-201 AC cases (nested params, `$.command`, URL paths, file paths, DB table/column) — no need for the extended grammar.
- `filelock` guards both `TaintState.save()` and `.load()` against the "rare concurrent-call" case the architecture doc flags (§6), using a sibling `<file>.lock` path.
- STORY-302's "sanitized output (post-masking, if applicable) is what's returned to the context window" doesn't require `post_tool_use` to do its own masking — masking already happened in `pre_tool_use` before the tool executed, so `post_tool_use` just passes `tool_response` through unchanged while updating taint flags.
- `run_scanner()` never raises regardless of what goes wrong (unknown scan type, subprocess spawn failure, malformed JSON stdout) — always returns `{"ok": False, "reason": ...}` on any failure, which `onboard()` then surfaces as a clear `OnboardError` message.

## Recent Decisions (2026-07-09, `*bloop *impl copyable system messages`)
- Suggested onboard command defaults to **`--list-type trust`**, not allowlist: `pre_tool_use`'s masking decision (`MaskingEngine.decide`) only ever reads `match.is_trusted`, never `match.is_sensitive` — so only a trust-list rule can actually prevent the same call from being masked again. An allowlist suggestion would look plausible but silently not work.
- JSONPath in the suggestion is `$.<payload_field>` (the field that was actually masked), not a separately-inferred "destination field" — matches Smith's handoff literally and avoids introducing a second per-tool config map (`DEFAULT_PAYLOAD_FIELDS` already exists; didn't add a parallel one).

- Detecting "did masking actually happen" reuses `MaskingEngine.apply_mask`'s existing no-op-returns-the-same-object behavior (`updated_input is tool_input`) rather than duplicating its internal validity checks in `hook_adapter.py` — kept the no-op contract in one place (DRY).

- Trin's UAT caught a real bug I missed: I `shlex.quote()`'d every dynamic argument in `_suggest_onboard_command` but hardcoded the `--target` placeholder as a raw string, forgetting `<`/`>` are shell metacharacters. **Lesson: when building a "copy-paste this" string, quote every single token uniformly — including static placeholders — not just the ones holding dynamic/untrusted data.
