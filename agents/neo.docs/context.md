- `MaskingEngine.apply_mask()` degrades to a no-op (returns args unchanged, logs a warning) rather than raising when `payload_field` is `None`, missing from `args`, or `args` isn't a dict — satisfies STORY-401's "masking never throws a runtime error" AC without needing a blanket try/except in the adapter.
- `hook_adapter.py`'s `DEFAULT_PAYLOAD_FIELDS` (`Write→content`, `Edit→new_string`, `Bash→command`, `WebFetch→prompt`) encodes the real Claude Code tool schemas (grounded, not guessed) but is injectable via the `payload_fields` param so callers/tests aren't locked to it.
- Audit log (`.scalene/audit.log`) writes only happen in `hook_adapter.py` and `onboard.py`, never `masking.py` — matches the architecture component diagram (only Adapter and Onboarding CLI touch `audit.log`).
- **Phase 3**: `LocalHeuristicChecker` (reputation.py) implements exactly the 3 heuristics architecture §7.4 named — IP-literal (via stdlib `ipaddress`), punycode (`xn--` label prefix), suspicious-length label (>30 chars) — no network calls anywhere in `src/scalene/` (grepped to confirm), satisfying "onboarding works fully offline."
- STORY-601 (scanner isolation): added a real `subprocess.run` boundary (`subprocess_isolation.run_scanner` → `scan_worker.py` as a `python -m` entrypoint) rather than just calling the checker/scanner functions in-process, because the story explicitly requires a separate process, not just logical isolation. `SCALENE_BYPASS=1` is set in the subprocess's env, and — symmetrically — `hook_adapter.pre_tool_use`/`post_tool_use` now check `os.environ.get("SCALENE_BYPASS") == "1"` and short-circuit (pass through unmodified / don't touch taint state) so the scanner's own actions can't recurse back through policy evaluation.

## Key Findings
- `jsonpath_ng.parse` (not `.ext`) is sufficient for all STORY-201 AC cases (nested params, `$.command`, URL paths, file paths, DB table/column) — no need for the extended grammar.
- `filelock` guards both `TaintState.save()` and `.load()` against the "rare concurrent-call" case the architecture doc flags (§6), using a sibling `<file>.lock` path.
- STORY-302's "sanitized output (post-masking, if applicable) is what's returned to the context window" doesn't require `post_tool_use` to do its own masking — masking already happened in `pre_tool_use` before the tool executed, so `post_tool_use` just passes `tool_response` through unchanged while updating taint flags.
- `run_scanner()` never raises regardless of what goes wrong (unknown scan type, subprocess spawn failure, malformed JSON stdout) — always returns `{"ok": False, "reason": ...}` on any failure, which `onboard()` then surfaces as a clear `OnboardError` message.

## Recent Decisions (2026-07-10, `*bloop *impl phase-1` — Secrets Scan Upgrade, STORY-801)
- Verified (docs, not assumption) that `detect-secrets` never makes network calls by default before adding it as a dependency — the `requests` transitive dependency it pulls in is only used by the opt-in `--only-verified` API-validation feature (Stripe/Twilio/IBM Cloud/etc. "is this secret still live" check), which Scalene will never invoke. Confirmed via GitHub README + PyPI/search cross-check. Satisfies the architecture's "no network egress introduced" requirement (`docs/ARCHITECTURE.md` §11.5) — this was a real risk worth checking, not a rubber-stamp.
- **Critical implementation finding**: `detect_secrets.core.scan.scan_line()` (the "adhoc string scan" API) is the wrong function to build on — its own source comments say it deliberately skips entropy-limit filtering for non-quoted matches ("we perform the limit filtering outside this function... to see *why* secrets have failed to be caught"), because it's meant to power `detect-secrets scan --string`'s raw-entropy CLI display, not a filtered yes/no answer. Naively using it flagged plain words in "just some ordinary markdown docs" as "Base64 High Entropy String" secrets. Caught this **before** writing any implementation by testing clean text first and getting a non-empty false-positive result — did not proceed until root-caused via the plugin source (`high_entropy_strings.py`'s `analyze_line`). Used `scan_file()` instead (writes the text to a real temp file first) — the fully filtered, production path used by the real CLI scan.
- Deduped exact-duplicate finding strings (`list(dict.fromkeys(findings))`) since multiple plugins can legitimately fire on the same secret (e.g. `Hex High Entropy String` + `Secret Keyword` both matching one generic `api_secret = "..."` assignment) — kept distinct-type findings (real signal), only dropped literal repeats.
- Added a new test (`test_detects_bare_provider_token_old_regexes_missed`, a bare `ghp_...` token with no keyword context) specifically to prove the upgrade's value — confirmed red against the old 3-regex implementation before touching `secrets_scan.py`, confirmed green after. Also added `test_never_attempts_network_access` (monkeypatches `socket.socket.connect` to raise) as a standing regression guard for the architecture's no-network-egress requirement, not just a one-time check.
- Kept `scan_text_for_secrets(text: str) -> list[str]`'s signature unchanged — `scan_worker.py` (its only caller) needed no changes at all. The temp-file write/cleanup is fully internal to the function.
- `detect-secrets` added as a normal (non-optional) `pyproject.toml` dependency, not an extra — confirmed `hook_adapter.py`/`cli.py` (the <15ms hot path) never import `secrets_scan`/`scan_worker` at all (grepped to verify), so this doesn't touch the hot-path import graph the way `textual` will need to avoid for the Sprint 2 console (E7).

## Recent Decisions (2026-07-09, `*bloop *impl copyable system messages`)
- Suggested onboard command defaults to **`--list-type trust`**, not allowlist: `pre_tool_use`'s masking decision (`MaskingEngine.decide`) only ever reads `match.is_trusted`, never `match.is_sensitive` — so only a trust-list rule can actually prevent the same call from being masked again. An allowlist suggestion would look plausible but silently not work.
- JSONPath in the suggestion is `$.<payload_field>` (the field that was actually masked), not a separately-inferred "destination field" — matches Smith's handoff literally and avoids introducing a second per-tool config map (`DEFAULT_PAYLOAD_FIELDS` already exists; didn't add a parallel one).
- Pattern defaults to an anchored, `re.escape`d exact literal of the real value (`^...$`) — narrowest possible rule, safest default; the command is shell-quoted (`shlex.quote`) per argument so it's copy-paste-safe even when the real value contains quotes/backslashes.
- Detecting "did masking actually happen" reuses `MaskingEngine.apply_mask`'s existing no-op-returns-the-same-object behavior (`updated_input is tool_input`) rather than duplicating its internal validity checks in `hook_adapter.py` — kept the no-op contract in one place (DRY).
- User asked for a repeatable e2e/smoke test instead of an ad-hoc verification script (rejected my `/tmp` one-off python invocation mid-task) — wrote `tests/test_onboard_suggestion_e2e.py` as a permanent test instead. **Lesson: don't reach for a throwaway shell/script verification step when a real test file achieves the same confidence and stays in the suite — default to writing the test, even for "just checking it works" moments.**
- Trin's UAT caught a real bug I missed: I `shlex.quote()`'d every dynamic argument in `_suggest_onboard_command` but hardcoded the `--target` placeholder as a raw string, forgetting `<`/`>` are shell metacharacters. **Lesson: when building a "copy-paste this" string, quote every single token uniformly — including static placeholders — not just the ones holding dynamic/untrusted data. The placeholder is exactly the part most likely to be pasted unedited.**
