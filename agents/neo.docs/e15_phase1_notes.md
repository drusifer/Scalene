# Neo — Sprint 9 (E15) Phase 1 implementation notes (2026-07-21)

## What shipped
- `scanner.py`: `load_scanners(raw_config)` — builds a registry from the 2 builtins + `scanners.extra:` entries (importlib-loaded, validated against the Scanner protocol, collision-checked). New `ScannerRegistryError` (avoids a circular import back into `policy_config.py`).
- `policy_config.py`: `PolicyConfig.scanners` field (defaults to `dict(SCANNERS)` — a bare `PolicyConfig()` is byte-identical to today). `from_yaml` parses `scanners:` first, builds the registry, then validates each rule's `scanner` field against *that* registry (not the module constant) before constructing `PolicyRule`. `PolicyRule.__post_init__`'s own scanner-name check is removed (documented in its docstring — moved, not silently dropped).
- `resource_verifier.py`: `evaluate()`/`decide_access()` now iterate `config.scanners`, not the module-level `SCANNERS` import.
- `onboard.py`: `identify_targets()`/`_onboard_resource()`/`onboard_targets()` all gain an optional `scanners` param (defaults to module `SCANNERS` — every existing call site unaffected). `main()` loads the project's real registry once (`PolicyConfig.from_yaml(policy_path).scanners`, falling back to builtins if no policy file exists yet) and threads it through.
- `cache_refresh_worker.py`: left on the module-level `SCANNERS` — see scope note below.

## Real implementation-time findings (not assumed away)
1. **A real regression, caught by an existing test, not invented for this pass**: moving `PolicyRule`'s scanner-name validation out of `__post_init__` (per §18.1) silently broke the *legacy* `onboard()`/`onboard_targets()` path — `_onboard_resource()` constructs a `PolicyRule` directly, never through `PolicyConfig.from_yaml`'s new validation loop, so a typo'd `--scanner` would have silently written an ineffective rule again (E12/STORY-1201's original defect, reintroduced). Fixed: `_onboard_resource()` now validates `scanner` against its own `registry` parameter directly, restoring the fail-loud behavior for the CLI/library path. `tests/test_onboard.py::test_invalid_rule_fields_reuse_policyrule_validation` (pre-existing) caught this — I didn't need to write a new test to find it, which is exactly the point of running the full suite before calling a phase done.
2. **Diagram-drift guard test caught my own architecture-writing getting ahead of implementation**: §18's class diagram (added during the architecture step) included `ReputationChecker`/`LocalHeuristicChecker`/`URLHausChecker`/`ReputationResult` — none of which exist in source yet (they're Phase 3 scope). `tests/test_architecture_docs.py` correctly failed on this. Reverted those diagram entries now; will re-add them as part of Phase 3's actual implementation, not before. This is the diagram-drift guard doing exactly its job — worth naming, since this project's retro backlog has flagged diagram drift 4+ times as a *missed* thing; this time the guard caught it before merge, not after.
3. **`cache_refresh_worker.py` scope decision, made and documented, not silently skipped**: it's a detached subprocess with no shared memory, so making it see a config-declared scanner would require passing `policy_path` across the subprocess boundary (through `_spawn_background_refresh()`/`refresh_if_needed()`, neither of which carries it today) — real plumbing with zero payoff since no real config-declared scanner ships this epic. Left on the module `SCANNERS`, with an explicit code comment explaining the gap and its (bounded, fail-safe) blast radius: a background refresh for such a resource would no-op (return 2) until the next lookup retries normally. Revisit when a real custom scanner ships.
